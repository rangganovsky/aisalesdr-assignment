### 1. Bulk Enrich — Backend

Add to `models.py`:
```python
class BulkEnrichRequest(BaseModel):
    lead_ids: list[int] = Field(..., min_length=1)  # FastAPI returns 422 for empty list

class BulkEnrichResponse(BaseModel):
    enriched: list[Lead]
    not_found: list[int]
```

New endpoint:

```python
POST /leads/bulk-enrich
Body: {"lead_ids": [1, 2, 3]}
Response: {
  "enriched": [Lead],    # all leads from lead_ids that exist (whether newly or already enriched)
  "not_found": [int]     # IDs that don't exist in the database
}
```

Implementation:
- Extract a private helper `_enrich_lead(lead: Lead) -> Lead` in main.py that both endpoints call:
  ```python
  def _enrich_lead(lead: Lead) -> Lead:
      lead.is_enriched = True
      lead.job_title = lead.job_title.title()
      return lead
  ```
- Refactor existing `POST /leads/{id}/enrich` to call `_enrich_lead(lead)` instead of inline logic
- Bulk enrich: build `lead_map = {lead.id: lead for lead in leads_db}` once per request, then call `_enrich_lead(lead_map[id])` for each valid ID
- Best-effort: enrich found IDs, report not-found IDs. Return 400 if lead_ids is empty.

### 2. Bulk Enrich — Frontend

- Add checkbox column to `LeadTable` (leftmost, before ID column)
- Track selected IDs with `useState<Set<number>>(new Set())`
  - IMPORTANT: React requires immutable updates. Always spread into a new Set:
    - Add: `setSelected(prev => new Set([...prev, id]))`
    - Remove: `setSelected(prev => new Set([...prev].filter(x => x !== id)))`
    - Toggle all: `setSelected(allVisible ? new Set(visibleIds) : new Set())`
    - Never: `selected.add(id); setSelected(selected)` — won't re-render
- "Select all" checkbox in header: toggles all currently visible/filtered rows
- When 1+ rows selected, show "Enrich Selected (N)" button in the filter bar area
- Clear selection (reset to empty Set) whenever filters change (prevents stale IDs from previous filter)
- Disable bulk enrich button when set is empty
- After bulk enrich: re-fetch table first, then show inline message "Enriched N leads (M not found)", then clear selection
  (re-fetch first ensures message counts reflect actual DB state, not just API response)
- Show loading spinner on bulk enrich button while in-flight

### 6. Part C Rewrite (`engineering_thinking.md`)

Rewrite to add depth. Priority order:
1. **WHY PostgreSQL**: mature JSONB (enrichment data is semi-structured), native full-text search, row-level security for future multi-tenancy, strong Python ecosystem (SQLAlchemy/asyncpg)
2. **WHY Celery**: distributed retries with exponential backoff, Flower dashboard for monitoring enrichment queue, scales horizontally — threads don't
3. **Tradeoffs (explicit)**: in-memory = data loss on Vercel cold start; seed data always reloads but user-added leads don't persist across cold starts; fix = Neon free tier (0.5GB, 30-min setup)
4. **90-day sequencing with rationale**: integration tests FIRST (enrichment pipeline: input → normalization → storage) to establish data quality trust; THEN CI/CD (you can't hire well without a process); THEN hire (Senior Fullstack, not Junior — no time for mentorship at seed stage)
5. **WHY Temporal over Celery Beat for campaigns** (if space): durable execution across days, workflow versioning, handles the case where a user edits a campaign mid-sequence

## Key Design Decisions

- **Bulk enrich: best-effort partial** — enrich found IDs, report not-found IDs in response. Don't fail the whole batch.
- **Select all = filtered/visible leads only** — respects the current filter state
- **Dict lookup for bulk enrich** — one full-table scan at request start, then O(1) per ID
